# Decision Log - Tokenised NASDAQ

This decision log records major design choices made during the development of Tokenised NASDAQ. Its purpose is to make trade-offs explicit rather than treating design choices as self-evident. I found that writing decisions in this way improved both the technical clarity of the project and my ability to think about market infrastructure as a set of strategic choices rather than only programming tasks.

| Decision | Options considered | Chosen option | Why chosen | Trade-off introduced |
|---|---|---|---|---|
| On-chain order matching | Off-chain matching engine with periodic on-chain settlement; fully on-chain matching and execution | On-chain matching | Provides transparent and deterministic rule enforcement at the point of execution, with directly auditable state transitions and fill events. | Higher gas consumption, lower throughput under network congestion, and tighter scalability limits than centralized matching. |
| Matching policy | Simpler matching logic with minimal ordering guarantees; explicit market microstructure priority rules | Price-time priority | Better reflects established market practice by prioritizing execution quality and fairness through an explicit, inspectable ordering policy. | Additional implementation complexity in order management and state updates, especially for partial fills and queue position handling. |
| Price/snapshot verification | Direct oracle-style price ingestion without verifiable snapshot commitments; committed snapshot model with proof-based verification | Merkle-proof verification | Improves auditability by anchoring entitlement or pricing datasets to a committed root, making verification explicit rather than implicit. | More complex operational workflow, including snapshot construction, proof generation, and off-chain data integrity requirements. |
| Settlement representation | Internal abstract balance ledger; token-based settlement using transferable contract standards | ERC-20 settlement | Provides a clear and composable settlement model that integrates cleanly with wallet tooling and smart-contract interfaces. | Simplifies legal settlement realism relative to traditional equity custody and introduces token UX dependencies (approve/execute flows). |
| Market mode design | Limit-order-only execution; dual execution modes combining limit order and instant buy | Dual mode (limit order + instant buy) | Balances market-structure discipline (limit orders) with user accessibility (instant buy), improving practical usability for varied participant intent. | Greater UI and backend complexity, including additional validation paths, error semantics, and execution-state communication. |
| Event indexing strategy | Chain-only query model per page request; persisted local indexer with rebuild capability | Persisted indexer plus rebuild path | Improves application responsiveness and historical query reliability by storing normalized activity for portfolio, transaction, and cashflow views. | Introduces synchronization risk, timeout handling requirements, and operational overhead to manage index state recovery. |
| Administrative control scope | Open administrative actions to any connected wallet; restricted actions to approved administrator wallets | Admin-wallet gating | Strengthens governance and operational safety by reducing unauthorized access to sensitive listing, lifecycle, and configuration actions. | Increased governance centralization and key-management dependence, with potential trust concerns if admin controls are not transparently managed. |
| Award/distribution tracking | Single aggregate reward metric; epoch/window-based mechanism with explicit period state and claims | Epoch/window reward logic | Produces clearer temporal accountability for participation and rewards by linking outcomes to explicit windows and claimable epochs. | More state-management complexity, including window timing drift, claim synchronization, and user-facing interpretation of epoch status. |

Overall, these decisions indicate that Tokenised NASDAQ is not only an engineering prototype but also a structured exploration of market-design trade-offs. The project demonstrates that architectural choices in financial infrastructure are simultaneously technical and strategic decisions with direct commercial implications. Documenting these choices in decision-log form improves both implementation discipline and investor-facing clarity, because it makes constraints, risks, and rationale inspectable.
